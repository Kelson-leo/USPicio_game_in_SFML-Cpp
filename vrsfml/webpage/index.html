VRSFML

an opinionated fork of SFML

A modern C++23 multimedia library with first-class web support via Emscripten, an immediate-mode drawing API in the spirit of Dear ImGui, highly-optimized even for low-end devices, and exceptionally fast to compile.

Browse on GitHub Try examples in your browser

What makes it different

Runs in the browser

Built-in Emscripten support. Every example compiles to WebGL2 and runs in a browser tab -- no #ifdefs, no shims, no extra build tooling on the user side.

Modern OpenGL only

No glBegin, no glMatrixMode. The pipeline targets OpenGL ES 3.1 with built-in shaders that work the same on desktop, mobile, and the web.

Safe by default

Invalid states are unrepresentable. Resources are created via factory functions returning sf::base::Optional. Default-constructed empty objects are gone.

Aggregates everywhere

Vec2, Rect2, Transformable, Vertex, RenderStates: all aggregates with public fields and designated initializers. No getter/setter dance.

Auto-batching

Sequential draw calls sharing the same texture, shader, and blend mode collapse into a single GPU draw call automatically. Your drawing code stays straightforward.

No more white-square bug

Sprite and Shape don't store a texture pointer. The texture is passed at draw time, so dangling-texture bugs become a compile error rather than UB.

Built-in ImGui

A first-class SFML::ImGui module replaces the out-of-tree imgui-sfml. One CMake option, ImGui works on every supported platform including the web.

Multiple audio devices

Play game audio on the speakers and voice chat on the headset, in the same process. Each sf::Sound is bound to a specific PlaybackDevice.

SDL3 backend

Native Wayland, smoother resize, expanded scancodes, and far better controller mapping than upstream's hand-rolled platform code. Free upgrade, same SFML feel.

Stack traces on every error

Assertion failures, lifetime-tracker fires, and library errors print a full call stack via vendored libbacktrace. No more guessing where things went wrong.

Immediate drawing API

Don't let OOP get in the way of drawing. Skip the sprite and draw a texture directly, or draw a shape without creating an intermediate object at all.

SFML or VRSFML -- which should I pick?

Both libraries are good at what they aim for; they aim at different things. Use this table to decide before reading further.

Dimension SFML 3.x VRSFML
Best for Beginners learning C++ and 2D graphics; small projects that want wide platform support today. Experienced C++ devs after modern APIs, web deployment, and aggressive performance.
C++ knowledge required Modest. C++17 features, forgiving API. Solid intermediate-to-advanced. C++23, designated initializers, optional-based control flow, aggregate idioms.
Stability Stable, semver-tracked releases. Public API rarely breaks within a minor version. Experimental. Used in shipped games but evolving -- expect occasional API breakage.
API style Classic OOP. Class hierarchies, virtual draw, getter / setter pairs. Multi-paradigm and data-oriented friendly. Aggregates with public fields, designated initializers; OOP available where it actually helps.
Error handling Exceptions; resources have empty default states. sf::base::Optional factories; no empty states; the public API does not throw.
Lifetime safety Manual. The "white-square problem" is documented behavior. Compile-time for sprites and shapes; runtime tracker for the rest of the resource graph in debug builds.
Web (Emscripten / WebGL) Not supported. First-class. Same source compiles native and WebGL2; every example runs in a browser tab.
Mobile (Android / iOS) Supported. Not yet shipped. The SDL3 backend should make porting straightforward -- contributions welcome.
Platform backend Hand-rolled per-OS code. SDL3. Battle-tested; native Wayland, better controller mapping, smoother resize.
OpenGL pipeline Mix of modern and legacy calls (glBegin, glMatrixMode, …). Modern OpenGL ES 3.1 only.
Drawable batching None. One draw call per drawable. Transparent auto-batching, plus manual DrawableBatch and TextureAtlas.
ImGui integration Out-of-tree imgui-sfml compatibility layer. Built-in SFML::ImGui module.
Multi-device audio One active playback device at a time. Multiple simultaneous playback devices.
Compile times Standard. Notably faster on clean rebuilds (custom vocabulary types, header hygiene, PImpl everywhere).
Reference docs Full Doxygen reference plus official tutorials. Migration guide and commented examples; reference docs are sparse.
Community & bindings Large, established. Many third-party language bindings (Python, Rust, …). Small. No third-party bindings yet -- reach out to the author for help.

Short version: pick SFML if you want the safer, better-documented, beginner-friendly option with mobile support today. Pick VRSFML if you want the web, modern OpenGL, auto-batching, and faster builds, and you're comfortable on the bleeding edge.

Compile times that don't ruin your day

VRSFML treats compile-time as a first-class concern. Catch-all headers are gone, every translation unit gets aggressive PImpl, and the public API uses sf::base vocabulary types instead of pulling in <optional>, <memory>, <type_traits>, or <iostream>.

The chart below is the scenario closest to a real edit-build inner loop: the SFML library is already compiled, and you rebuild six small example programs (island, shader, stencil, tennis, keyboard, joystick) from scratch.

726 ms

288 ms

2.5x faster wall time, 2.7x less compiler CPU work per file. Six TUs each, same source set, debug build, lld, no ccache, 16 jobs.

Full picture, four scenarios

Selective rebuild: third-party deps (freetype, vorbis, ogg, flac, harfbuzz, imgui, …) are kept on disk between iterations, so the measurement reflects only SFML's own code plus the chosen examples and tests -- the workflow you actually have when iterating on a project.

Library + examples 128 / 209 4.55 s 2.23 s 2.04x

Six user TUs from scratch 6 / 6 726 ms 288 ms 2.52x

TUs column shows SFML / VRSFML. VRSFML compiles more translation units in the first three rows (extra SFML::ImGui module, more bundled examples, larger test suite), and is still ~2x faster overall -- meaning per-TU compiler work is roughly 3-4x lower. The fourth row, where both projects compile the same six user-side TUs, isolates that effect cleanly.

Methodology: hyperfine 1 warmup + 3 timed runs, clang++, debug build, -fuse-ld=lld, BUILD_SHARED_LIBS=OFF, SFML_USE_SYSTEM_DEPS=OFF, ccache disabled, -j 16, ninja. Between iterations only SFML-owned object files and archives are deleted; bundled deps stay built. Replace with your own measurements on your own hardware.

  • No SFML/Graphics.hpp-style catch-all headers.
  • Custom vocabulary types with minimal template depth.
  • In-place PImpl (no allocation, hides implementation).
  • Compiler built-ins for traits and index sequences.
  • extern template for hot instantiations.

Bring your own batteries: SFML::Base

VRSFML ships a small standalone utilities library used internally and re-exported for users. Compile-time-friendly, allocation-conscious, and independent from the C++ standard library where it matters.

Optional · UniquePtr · Variant Vocabulary types that don't pull in <optional> / <memory> / <variant>.
String · StringView · Span · Vector Drop-ins for the standard equivalents, designed for fast compiles and good debug perf.
InPlaceVector<T, N> Capacity-bounded, stack-only, no allocation. Ideal for sound pools and bounded queues.
SmallVector<T, N> Inline storage with heap fallback. The right default when most cases stay small.
ChunkedVector<T> Stable references on growth. Append-only structures, indexed entity arenas.
FixedFunction<Sig, N> Allocation-free std::function: storage lives inline, no heap, no vtable surprise.
InPlacePImpl<T, N> PImpl that hides the implementation without allocating. Used everywhere internally.
ThreadPool Fixed-worker pool. No <thread> in your headers.
EnumArray · EnumClassBitwiseOps Strongly-typed lookup tables and bitflag enums without macro gymnastics.
MiniPFR Aggregate reflection -- iterate fields by index, get field names, build SoA containers from any struct.

See the rockets example for a struct-of-arrays container generated from a plain aggregate via MiniPFR.

Features upstream couldn't accept

VRSFML exists because some changes felt too disruptive for upstream's roadmap. Each of the following was proposed, discussed, and ultimately shelved or reverted on master. They all ship here.

  1. Modern OpenGL ES backend, with Emscripten support -- deferred to a hypothetical future SFML 4.x.
  2. Factory APIs returning Optional, no empty states -- reverted in #3152.
  3. Sprites and shapes without a stored texture pointer -- rejected in #3072 and #3080.
  4. Cross-resource lifetime tracking -- rejected in #3097.
  5. Removal of sf::VertexArray in favor of std::vector<Vertex> -- rejected in #3118.
  6. Multiple simultaneous audio playback devices -- deemed out-of-scope.
  7. Doctest as the testing framework -- replaced with Catch2.
